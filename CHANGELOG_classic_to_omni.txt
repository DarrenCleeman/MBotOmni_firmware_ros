This document details every change made to convert the MBot Classic firmware
(2-wheel differential drive) into the MBot Omni firmware (3-wheel
omnidirectional drive at 120-degree spacing) for ROS2 Jazzy.

4. CORE FIRMWARE (Main Application Logic)
   File: src/mbot_classic_ros.c

4.1 Forward Kinematics - Body Velocity Calculation
    OLD (differential):
      static void mbot_calculate_diff_body_vel(
          float wheel_left_vel, float wheel_right_vel,
          float* vx, float* vy, float* wz)
      {
          *vx = DIFF_WHEEL_RADIUS * (wheel_left_vel - wheel_right_vel) / 2.0f;
          *vy = 0.0;
          *wz = DIFF_WHEEL_RADIUS * (-wheel_left_vel - wheel_right_vel)
                / (2.0f * DIFF_BASE_RADIUS);
      }

    NEW (omnidirectional):
      static void mbot_calculate_omni_body_vel(
          float wheel_left_vel, float wheel_right_vel, float wheel_back_vel,
          float* vx, float* vy, float* wz)
      {
          *vx = OMNI_WHEEL_RADIUS * (wheel_left_vel * INV_SQRT3
                                   - wheel_right_vel * INV_SQRT3);
          *vy = OMNI_WHEEL_RADIUS * (-wheel_left_vel / 3.0f
                                   - wheel_right_vel / 3.0f
                                   + wheel_back_vel * (2.0f / 3.0f));
          *wz = OMNI_WHEEL_RADIUS * -(wheel_left_vel + wheel_right_vel
                                    + wheel_back_vel)
                / (3.0f * OMNI_BASE_RADIUS);
      }

    Key differences:
    - Takes 3 wheel velocities instead of 2
    - vy is now computed (was always 0 in differential)
    - Uses sqrt(3) factors for 120-degree wheel geometry
    - Angular velocity uses sum of all 3 wheels

4.2 Inverse Kinematics - Motor Target Velocities
    OLD (MODE_MBOT_VEL case):
      float target_vel_left  = (local_cmd.vx - DIFF_BASE_RADIUS * local_cmd.wz)
                               / DIFF_WHEEL_RADIUS;
      float target_vel_right = (-local_cmd.vx - DIFF_BASE_RADIUS * local_cmd.wz)
                               / DIFF_WHEEL_RADIUS;

    NEW (MODE_MBOT_VEL case):
      float target_vel_left  = (SQRT3 / 2.0f * local_cmd.vx
                              - 0.5f * local_cmd.vy
                              - OMNI_BASE_RADIUS * local_cmd.wz)
                              / OMNI_WHEEL_RADIUS;
      float target_vel_right = (-SQRT3 / 2.0f * local_cmd.vx
                              - 0.5f * local_cmd.vy
                              - OMNI_BASE_RADIUS * local_cmd.wz)
                              / OMNI_WHEEL_RADIUS;
      float target_vel_back  = (local_cmd.vy
                              - OMNI_BASE_RADIUS * local_cmd.wz)
                              / OMNI_WHEEL_RADIUS;

    Key differences:
    - Computes 3 target velocities instead of 2
    - Supports vy (lateral) commands
    - Uses 120-degree geometry with sqrt(3) factors

4.3 Hardware Initialization (mbot_init_hardware)

    MODE_MOTOR_VEL (wheel velocity control):

    MODE_MBOT_VEL (body velocity control):

8. ODOMETRY
   File: src/mbot_odometry.c
================================================================================

    NO CHANGES REQUIRED.

    The odometry function already supported full holonomic motion:
      void mbot_calculate_odometry(float vx, float vy, float wz, float dt,
                                   float* x, float* y, float* theta)
      {
          *x     += vx * dt * cos(*theta) - vy * dt * sin(*theta);
          *y     += vx * dt * sin(*theta) + vy * dt * cos(*theta);
          *theta += wz * dt;
      }

    The vy parameter was previously always 0.0 (differential drive had no
    lateral velocity). With the omni port, vy is now a real computed value
    from the forward kinematics.

9.1 Motor Polarity Detection
    OLD (differential, 4 combinations for 2 motors):
      float gyro_z[4] = {0, 0, 0, 0};
      float motor_duties[4][2] = {
          { spd,  spd},    // both forward
          {-spd,  spd},    // left back, right forward
          { spd, -spd},    // left forward, right back
          {-spd, -spd}     // both back
      };
      // Test each combo, read gyro, use complex switch to determine polarity

    NEW (omni, 8 combinations for 3 motors = 2^3):
      float gyro_z[8] = {0};
      float motor_duties[8][3] = {
          { spd,  spd,  spd},     // all positive
          {-spd,  spd,  spd},     // left negative
          { spd, -spd,  spd},     // right negative
          { spd,  spd, -spd},     // back negative
          {-spd, -spd,  spd},     // left+right negative
          {-spd,  spd, -spd},     // left+back negative
          { spd, -spd, -spd},     // right+back negative
          {-spd, -spd, -spd}      // all negative
      };
      // Find max positive gyro_z (CCW rotation)
      // Negate the signs that caused CCW:
      params.motor_polarity[MOT_L] = (motor_duties[rot_idx][0] > 0) ? -1 : 1;
      params.motor_polarity[MOT_R] = (motor_duties[rot_idx][1] > 0) ? -1 : 1;
      params.motor_polarity[MOT_B] = (motor_duties[rot_idx][2] > 0) ? -1 : 1;

    The polarity detection logic was simplified: instead of a switch statement,
    it uses a direct sign-inversion formula.

16. SUMMARY
================================================================================

    The port touches every layer of the firmware stack:

    Layer               What Changed
    ------------------  -------------------------------------------------------
    Hardware config     Motor slots, physical parameters, math constants
    Kinematics          Forward/inverse equations (diff -> omni 120-degree)
    Motor control       2 motors -> 3 motors in all control modes
    PID control         Added back wheel controller + parameter server entries
    FRAM parameters     Added back_wheel_vel_pid, expanded arrays to size 3
    Calibration         8-combo polarity detection, 3-motor slope calibration
    ROS publishing      Back motor velocity published
    Build system        Targets renamed, lidar target removed
    Diagnostics         All prints updated for 3 motors

    The core design philosophy (state machine, control modes, FRAM persistence,
    ROS parameter server, dual CDC USB) was preserved. The changes are purely
    additive in terms of motor count and kinematic model.
